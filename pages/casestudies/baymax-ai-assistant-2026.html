Building a Portfolio AI Assistant: Technical Design and Model Choices

The motivation was straightforward. A recruiter visiting a portfolio has specific questions: what cloud platforms does this person know, have they worked with Kafka, what did they actually build at their last job? The answers are on the site, but scattered across pages. Navigating a resume page and two articles to piece together an answer takes effort most people won't spend. An AI assistant that already knows everything about me and can answer directly is a simpler experience.

That is what Baymax is. A small robot in the 3D room and a chat widget on every page, both backed by the same API: send a question, get a specific answer about my background.

How it works

The assistant runs as a separate Vercel service. Every question goes through the same path: a rate limit check using Upstash Redis (to protect the API key from abuse), then a cache lookup keyed to a normalised hash of the question. On a cache hit, the answer comes back immediately. On a miss, the question goes to Groq with a curated knowledge base attached as context, the response is stored in Redis with a 24-hour TTL, and the answer is returned.

The knowledge base is a single markdown file: my work history, certifications, projects, skills, and writing. No retrieval, no chunking, no vector database. The whole file is sent with every request as part of the system prompt.

Why Llama 3.3 70B on Groq

The model choice came down to two requirements: the knowledge base needed to fit in the context window, and the answers needed to be good enough to actually be useful.

Groq's Llama 3.3 70B Versatile has a 128k token context window. My knowledge base is well under 10,000 tokens. That means the entire thing fits in a single call without any retrieval logic. Smaller models on Groq (the 8B instant variants) only handle 8k context, which would either truncate the knowledge base or require chunking and retrieval, adding complexity for no real benefit at this scale.

The 70B parameter count matters for quality. Smaller models give noticeably vaguer answers to open-ended questions like "what does Jonas's experience with data governance look like?" A 70B model handles those with enough nuance to be genuinely useful. Groq's LPU hardware makes the 70B model fast enough that response time is rarely over two seconds, which is acceptable for an assistant on a portfolio.

What Redis is actually doing

Redis is doing two things here. First, rate limiting: each IP address is capped at 20 requests per minute. Without this, the API key is exposed on a public endpoint and anyone can drain the free tier quota. Second, response caching: recruiter questions repeat. The same handful of questions come up over and over, and there is no reason to call Groq each time. After the first call, the answer is stored against a hash of the normalised question and served instantly to anyone who asks the same thing.

The cache is also the clearest signal of whether the assistant is being used. A high HIT rate means the same questions are being asked repeatedly, which tells you something about what people actually want to know.

Something to think about

The knowledge base is a file I maintain manually. It is accurate today, but it will drift as my resume changes. The honest next step is a pipeline that fetches the portfolio site periodically and rebuilds the knowledge base automatically, so the assistant stays current without manual intervention. That is the gap between a working demo and something production-ready, and it is a smaller gap than it might seem.
